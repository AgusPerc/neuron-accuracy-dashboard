Glosario
Referencia rapida para explicar cada metrica y concepto del dashboard
Metricas principales
Accuracy (precision global)
De cada 100 llamadas, cuantas fueron clasificadas correctamente. Si un modelo tiene 65% accuracy, significa que 65 de cada 100 labels son correctos y 35 son errores.
Ejemplo: "Retell acierta en 65 de cada 100 llamadas del dataset limpio."
Precision (por label)
Cuando el modelo dice "promised_payment", que tan seguido tiene razon. Precision alta = pocas falsas alarmas. Si precision de promised_payment es 100%, significa que cada vez que Retell dice "el cliente prometio pagar", efectivamente lo hizo.
Ejemplo: "Cuando Retell dice que hay promesa de pago, tiene razon el 100% de las veces."
Recall (por label)
De todas las llamadas que realmente son "refused", cuantas detecta Retell. Recall alto = no se le escapan casos. Si recall de refused es 62.5%, significa que Retell detecta solo 5 de cada 8 rechazos — se le escapan 3.
Ejemplo: "De cada 8 rechazos reales, Retell solo detecta 5. Los otros 3 los clasifica como otra cosa."
F1 Score
Combinacion de precision y recall en un solo numero. F1 alto = el modelo es bueno tanto en detectar como en no dar falsas alarmas. Va de 0 (pesimo) a 1 (perfecto). Arriba de 0.90 es excelente, 0.80-0.90 es bueno, abajo de 0.80 necesita mejora.
Ejemplo: "El F1 de promised_payment es 0.97, o sea excelente. El de refused es 0.77, necesita mejora."
Fleiss Kappa
Mide que tanto coinciden multiples evaluadores entre si, corrigiendo por coincidencia aleatoria. Si 3 personas tiran una moneda, a veces coinciden por suerte — kappa descuenta eso. Un kappa en el rango 0.61-0.80 (donde estamos) significa que los 3 labelers coinciden significativamente mas que por azar.
0-0.20 = leve · 0.21-0.40 = aceptable · 0.41-0.60 = moderado · 0.61-0.80 = substancial · 0.81-1.0 = casi perfecto
Labels (categorias de llamada)
promised_payment
El cliente se comprometio a pagar una cantidad en una fecha especifica. Es el label mas importante — determina forecast de cobranza.
no_answer
La llamada no conecto con el deudor. Nadie contesto, o el telefono sono y colgo sin interaccion.
contacted
Alguien contesto y hubo interaccion breve, pero sin resultado concreto (no prometio pagar, no rechazo, no pidio callback).
voicemail
La llamada fue al buzon de voz / contestadora automatica.
callback_requested
El deudor pidio que le llamen despues (esta ocupado, quiere consultar, etc).
refused
El deudor rechazo pagar explicitamente ("no voy a pagar", "no me importa").
other
Cajón de sastre — la llamada no encaja en ninguna otra categoria. Un "other" alto indica que el modelo no sabe clasificar bien.
Metodologia
Claude como juez
Usamos Claude Opus 4.6 (el modelo mas avanzado de Anthropic, 1M tokens de contexto) para leer cada transcript completo y asignar el label correcto. Esos labels se tratan como "verdad" contra la que medimos a Retell y Llama. No es un humano, pero es el mejor juez automatizado disponible.
Retell / GPT-4 (produccion)
El modelo que actualmente clasifica las llamadas en produccion. Es el que estamos evaluando — queremos saber si etiqueta bien.
Llama 4 Scout (auditor)
Modelo de Meta, corrido via Groq (gratis). Sirve como segunda opinion independiente. Si Llama y Claude coinciden pero Retell no, es probable que Retell se equivoco.
